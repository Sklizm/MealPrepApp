# 2026-05-11 — Phase 2.5: DB polish before app integration

Pre-app cleanup pass. Goal was to close the few gaps that would otherwise have leaked into the .NET side, while explicitly leaving the auth/users layer alone (Codrin plans a later non-secured rebuild for personal end use).

## What was added

- **`Database/seeds/ingredients_seed.sql`** — 44 common ingredients (pantry / oils / dairy / produce / proteins / herbs), `MERGE`d on `Name`. `DefaultUnitID` is looked up by unit abbreviation against `[[Units]]` so the file stays human-readable.
- **`Database/10_phase25_additions.sql`** — two missing FK indexes (`IX_Ingredients_DefaultUnitID`, `IX_RecipeIngredients_UnitID`) + `RowVersion ROWVERSION NOT NULL` on `[[Recipes]]`. All idempotent.
- **`sp_GetIngredientUsage`** in `procs/04_ingredients.sql` — returns `RecipeCount` so the app can show "in use by N recipes" before attempting an FK-blocked delete.

## What was rewritten

- **`sp_UpdateRecipe`** now requires `@RowVersion BINARY(8)`. Reads current `RowVersion` after ownership check; if it doesn't match, `THROW 50004, N'Stale row — recipe was modified by another session', 1;`. The token is returned by `sp_GetRecipeFull` so the round-trip is the obvious one.
- **`sp_GetRecipeFull`** first result set now includes `RowVersion`.
- **`sp_FindRecipesByIngredients`** rewritten as one `GROUP BY r.RecipeID` over a `LEFT JOIN` to the `@IngredientIDs` TVP, instead of two `CROSS APPLY` subqueries. Same output shape; one scan of `RecipeIngredients` per recipe instead of two.

## Error code reservation

- **50004** — stale row (optimistic concurrency conflict). Joins 50001 (password reused), 50002 (not authorized), 50003 (not found).

## Build hiccup worth remembering

First attempt to `docker cp Database MealPrepDB:/tmp/Database` copied INTO the existing `/tmp/Database/` directory (left over from Phase 1/2), producing `/tmp/Database/Database/...` while sqlcmd ran the stale outer copy. Visible symptom: build "succeeded" but no new objects appeared.

Fix: `docker exec -u 0 MealPrepDB rm -rf /tmp/Database` (needs root since the existing files were owned by uid 1000) before the next `cp`. Worth folding into the standard rebuild step.

## SQL Server gotcha worth remembering

First attempt at the rewritten `sp_FindRecipesByIngredients` used `SUM(CASE WHEN ri.IngredientID IN (SELECT ID FROM @IngredientIDs) THEN 1 ELSE 0 END)`. SQL Server rejects this — **subqueries can't appear inside aggregate functions** (Msg 130). The working pattern is to `LEFT JOIN` the TVP and aggregate over the join match flag (`SUM(CASE WHEN m.ID IS NOT NULL THEN 1 ELSE 0 END)`).

## Verification (all green)

- FK indexes both present
- `RowVersion` column exists, type `timestamp`
- `dbo.Ingredients` count = 44
- `sp_GetIngredientUsage` returns 0 for unused, empty rowset for non-existent ID
- `sp_FindRecipesByIngredients` runs against empty TVP and returns no rows
- RowVersion round-trip: create → update with token → succeeds, token changes; second update with stale token → THROW 50004 with the right message

## Out of this pass (deliberate)

- No new user/auth procs. The auth layer stays frozen at Phase 2's shape; will likely be stripped wholesale in a later rebuild for end-user use.
- No DataGrip-only views, no `sp_GetAuditForUser`, no soft delete. These remain on TODO "Soon"/"Maybe Later".
