## Decisions made (logged in [[Decisions Log]])
- Core scope only — no meal plans / shopping lists / nutrition for v1
- Ingredients are global (no UserID) for v1
- Only one cascade: Recipes → RecipeIngredients
- UTC timestamps everywhere
- Idempotent scripts (re-runnable safely)

## Verification (same session)
Ran `run_all.sql` inside the `MealPrepDB` Docker container with `sqlcmd`. Results:
- All 6 tables exist; 6 PKs, 6 FKs, 6 UNIQUEs, 5 CHECKs, 2 DEFAULTs, plus the 3 expected non-PK indexes
- Seeds inserted: 12 [[Units]], 6 [[Categories]]
- Smoke test (insert user + ingredients + recipe + recipe_ingredients) round-trips correctly
- All CHECK constraints reject bad data (Quantity ≤ 0, negative times, Servings ≤ 0, UnitType not in {weight, volume, count})
- UNIQUE on `(RecipeID, IngredientID)` blocks duplicate ingredient rows
- Cascade on Recipes → RecipeIngredients works
- Deleting an Ingredient still in use is correctly blocked (RESTRICT)
- Test data cleaned up; all user tables back to 0 rows
