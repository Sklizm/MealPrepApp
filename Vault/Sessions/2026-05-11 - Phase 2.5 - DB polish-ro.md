# 2026-05-11 — Faza 2.5: Slefuire DB inainte de integrarea cu aplicatia

Pas de curatare pre-aplicatie. Obiectivul a fost sa inchidem cele cateva goluri care altfel s-ar fi scurs in partea .NET, lasand explicit stratul auth/users in pace (Codrin planuieste un rebuild ulterior nesecurizat pentru uz personal final).

## Ce s-a adaugat

- **`Database/seeds/ingredients_seed.sql`** — 44 de ingrediente comune (camara / uleiuri / lactate / produse / proteine / ierburi), facut `MERGE` pe `Name`. `DefaultUnitID` este cautat dupa abrevierea unitatii in [[Units-ro]] astfel incat fisierul ramane lizibil pentru oameni.
- **`Database/10_phase25_additions.sql`** — doi indecsi FK lipsa (`IX_Ingredients_DefaultUnitID`, `IX_RecipeIngredients_UnitID`) + `RowVersion ROWVERSION NOT NULL` pe [[Recipes-ro]]. Toate idempotente.
- **`sp_GetIngredientUsage`** in `procs/04_ingredients.sql` — returneaza `RecipeCount` astfel incat aplicatia sa poata afisa "in uz de N retete" inainte de a incerca o stergere blocata de FK.

## Ce s-a rescris

- **`sp_UpdateRecipe`** acum cere `@RowVersion BINARY(8)`. Citeste `RowVersion` curent dupa verificarea de proprietate; daca nu se potriveste, `THROW 50004, N'Stale row — recipe was modified by another session', 1;`. Token-ul este returnat de `sp_GetRecipeFull` deci round-trip-ul este evident.
- **`sp_GetRecipeFull`** primul set de rezultate include acum `RowVersion`.
- **`sp_FindRecipesByIngredients`** rescris ca un singur `GROUP BY r.RecipeID` peste un `LEFT JOIN` la TVP-ul `@IngredientIDs`, in loc de doua subquery-uri `CROSS APPLY`. Aceeasi forma de output; un singur scan al `RecipeIngredients` per reteta in loc de doua.

## Rezervare cod de eroare

- **50004** — rand invechit (conflict de concurenta optimista). Se alatura lui 50001 (parola reutilizata), 50002 (neautorizat), 50003 (negasit).

## Hiccup de build de retinut

Prima incercare de `docker cp Database MealPrepDB:/tmp/Database` a copiat IN directorul `/tmp/Database/` existent (lasat de la Faza 1/2), producand `/tmp/Database/Database/...` in timp ce sqlcmd rula copia exterioara invechita. Simptom vizibil: build-ul "a reusit" dar nu au aparut obiecte noi.

Corectare: `docker exec -u 0 MealPrepDB rm -rf /tmp/Database` (nevoie de root deoarece fisierele existente erau detinute de uid 1000) inainte de urmatorul `cp`. Util de inclus in pasul standard de rebuild.

## Capcana SQL Server de retinut

Prima incercare la rescrierea `sp_FindRecipesByIngredients` a folosit `SUM(CASE WHEN ri.IngredientID IN (SELECT ID FROM @IngredientIDs) THEN 1 ELSE 0 END)`. SQL Server respinge asta — **subquery-urile nu pot aparea in interiorul functiilor de agregare** (Msg 130). Pattern-ul care functioneaza este sa faci `LEFT JOIN` pe TVP si sa agreghezi peste flag-ul de potrivire al join-ului (`SUM(CASE WHEN m.ID IS NOT NULL THEN 1 ELSE 0 END)`).

## Verificare (toate verzi)

- Ambii indecsi FK prezenti
- Coloana `RowVersion` exista, tipul `timestamp`
- Numarul `dbo.Ingredients` = 44
- `sp_GetIngredientUsage` returneaza 0 pentru neutilizat, rowset gol pentru ID inexistent
- `sp_FindRecipesByIngredients` ruleaza pe TVP gol si returneaza fara randuri
- Round-trip RowVersion: create → update cu token → reuseste, token-ul se schimba; al doilea update cu token invechit → THROW 50004 cu mesajul corect
